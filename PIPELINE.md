# Sports News Blog — 스포츠 뉴스 자동화 파이프라인 v0.2

> **이 문서 하나로 처음 보는 사람도 클론 → 로컬 검증 → 라이브 전환까지 전부 실행할 수 있습니다.**
> 외부 시크릿 없이 `npm install && npm run pipeline:dry-run` 만으로 전체 파이프라인이 가짜 데이터(fixture)로 돌아가고, 5개 합격 기준 + 2개 차별점을 자동으로 검증합니다.
>
> **이 프로젝트가 하는 일 (한 문장):** 매일 여러 스포츠 사이트에서 그날의 이슈를 모아, **같은 사건을 다룬 2~3개 출처를 AI로 "합성"해 우리만의 원본 기사(한국어+영어)** 를 쓰고 → 블로그에 게시 + **카드뉴스 이미지**로 만들어 텔레그램 그룹방에 배포합니다. 브랜드명은 언급하지 않고 순수 스포츠 정보만 제공합니다.
>
> **🎯 사업 모델 (왜 만드는가):** 타깃은 **기사를 일일이 찾아볼 시간이 없는 바쁜 현대인**. 가치 사다리는 — ① 채팅방에서 종목별 **요약 카드뉴스**를 무료로 받아보고 → ② 더 궁금하면 **블로그 상세 기사**로 들어와(트래픽 발생) → ③ **구독 회원**에게는 경기 전 인텔리전스 카드·개인화 팔로우를 **이메일+텔레그램**으로 배달 → ④ 그 과정에서 **이메일·관심사 데이터**를 수집 → ⑤ 이용자·트래픽이 쌓이면 **온라인 배팅 사이트 배너 광고**로 직접 수익화. 즉 콘텐츠는 *고관여 스포츠 오디언스를 모으는 무료 미끼*이고, 진짜 제품은 그 오디언스(+데이터)에 대한 광고 도달 면입니다. (현재 MVP는 ①②③의 텔레그램 배달까지 구현; 이메일·데이터 스토어·배팅 배너는 후속.)
>
> **🔑 v0.2 차별점 (왜 우리 채팅방에 머무는가):** "뉴스 모아주기"는 commodity라 그것만으론 약합니다. 그래서 뉴스사이트가 *못 하는* 두 가지를 얹었습니다 —
> - **① 경기 전 인텔리전스 카드** — 다가올 경기별 폼·부상·상대전적을 1장으로, "무엇을 볼지/무슨 의미인지" (뉴스는 '벌어진 일'만 보도).
> - **② 개인화 팔로우** — "토트넘 팔로우" → 그 팀/선수 것만 골라 개별 푸시. firehose를 *내 피드*로 (리텐션 핵심).
>
> **v0.2 추가:** 라이브 LLM 어댑터(Anthropic) 구현 — Anthropic 키만으로 실제 Claude 합성·사실검증을 끝까지 돌려볼 수 있음(`pipeline:run`, 나머지는 fixture/mock 자동 대체).
>
> **MVP 범위:** 채팅방 요약 카드뉴스 + 블로그 상세 + 텔레그램 1채널 + ①경기전인텔 + ②개인화(텔레그램 배달). **후속(미구현):** 이메일 채널 + 구독자 데이터 수집(이메일·관심사), 온라인 배팅 사이트 배너 광고, 전 종목 커버리지, 카카오톡, 대화형 AI 봇.

## 1. 인프라 구조 (왜 각 요소가 필요한가)

```
매일 00:00 UTC — GitHub Actions가 자동으로 깨어남
        ↓
  [뉴스 사이트들] ── 그날의 스포츠 이슈 (원재료)
        ↓  수집(스크래핑)
  [이 프로젝트(Node.js)] ── 모으고 → 합치고(합성) → 검증 → 카드 제작
        ↓  발행
  [WordPress] ── 블로그 글 저장·노출  +  [Telegram] ── 카드뉴스 배포
        ↓
  독자에게 도달
```

| 요소 | 역할 | 비유 | 비용 |
|------|------|------|------|
| **Node.js 20+** | 파이프라인 실행 엔진 | 공장 라인 | 무료 |
| **GitHub Actions** | 매일 정해진 시각 자동 실행 + 시크릿 보관 | 자동 타이머 | 무료(Public 레포) |
| **Serper / NewsAPI** | "오늘 뭐가 이슈인지" 뉴스 URL 발견 | 신문 가판대 | 무료 티어 |
| **Playwright + robots.txt** | 허용된 페이지의 본문만 수집 | 예의 바른 수집원 | 무료 |
| **Anthropic(Claude)** | 다중 출처를 합성해 원본 기사 작성 + 사실 추출 | 기자 | 글당 과금 |
| **OpenAI·Gemini** | 자문 품질 평가(톤·가독성) — *발행 차단 권한 없음* | 교열 의견 | 소액/무료 |
| **sharp** | 카드뉴스 이미지 렌더링(로컬, API 불필요) | 디자인 템플릿 | 무료 |
| **WordPress + Polylang + Rank Math** | 블로그 글 저장 / 한·영 번역쌍 연결 / SEO 메타 | 출판사 | 호스팅비 |
| **Telegram Bot** | 카드뉴스 그룹방 배포 + 운영 알림 | 배달원 | 무료 |

> **왜 "스크래핑 + 합성"인가, 그리고 왜 그게 위험한가?**
> 남의 기사를 **한 개만 베껴 다시 쓰는 것**은 저작권·"hot news" 침해 위험이 큽니다. 어떤 코드 검사로도 이 위험을 완전히 없앨 수 없습니다. 그래서 이 시스템의 **1차 방어선은 "다중 출처 합성"** — 같은 사건을 보도한 2~3개 출처의 *사실*을 종합해 우리 구조·문장으로 **새 기사**를 씁니다. 이것이 변형적(transformative)이라 방어 가능하고, 부수적으로 사실 검증과 중복 병합에도 도움이 됩니다.

## 2. 핵심 개념 — 이 파이프라인이 지키는 5가지 약속(합격 기준)

이 프로젝트는 "그냥 돌아간다"가 아니라 **매일 다음 5가지가 충족돼야 성공**으로 간주하며, 각각을 자동 테스트로 검증합니다.

| # | 약속 | 어떻게 보장하는가 | 검증 테스트 |
|---|------|-------------------|------------|
| **AC#1** | 한국어+영어 기사가 자동 생성·발행된다 | WordPress에 KO·EN 각각 발행 + Polylang 번역쌍 연결 | dry-run: 이슈당 서로 다른 KO/EN 글 ID 기록 |
| **AC#2** | **표절도 없고 사실 오류도 없다** | 표절 게이트(유사도) + **사실 게이트(출처 대조)** 가 막음 | `factgate.seeded.test.mjs`: 출처가 2-1인데 기사가 3-0이면 **차단** |
| **AC#3** | 카드뉴스가 텔레그램에 가되 **깨진 이미지는 안 간다** | 렌더 후 글자 픽셀 검사(glyph-smoke) 통과해야 전송 | dry-run: PNG 유효 + 글자 렌더 확인 + 빈 이미지 아님 |
| **AC#4** | **같은 기사를 두 번 발행하지 않는다** | 사건 지문 + 발행 원장으로 중복 제거 | `dedup.idempotent.test.mjs`: 같은 날 2회 실행 → 2회차 0건 |
| **AC#5** | **사람 손 없이** 매일 끝까지 돈다 | cron + 모든 실패가 "조용히 넘어가지 않고" 알림 | dry-run: 프롬프트 0회로 완주 + 폴백마다 알림 |

> **가장 중요한 설계: 사실 검증은 "AI에게 맞냐고 물어보기"가 아니다.** 세 AI에게 "이 기사 정확해?"라고 물으면 자신만만한 환각을 통과시킵니다. 그래서 사실 게이트는 기사 속 숫자·이름·인용을 **출처 원문에 직접 대조**하는 결정론적 방식입니다. (예: 스코어 `3-0`은 출처의 `13-0` 안에 들어 있어도 매칭되지 않음 — 정수 전체일치)

## 3. 수집 소스 — "오늘의 뉴스를 어디서, 어떻게 가져오는가"

### 무엇을 하는가
종목별로 그날의 주요 이슈를 찾아, **합법적으로 가장 안전한 순서**로 원재료를 모읍니다.

```
1. Serper(구글 뉴스 검색) — "오늘 이 종목에서 뭐가 이슈인지" URL을 발견
        ↓
2. robots.txt 검사 — 각 사이트가 자동 수집을 허용하는지 확인 (robots-parser)
        ↓  허용된 페이지만
3. Playwright — 본문 fetch (불허 시 스니펫/제목만 사용)
        ↓
원재료: { 종목, 매체, URL, 제목, 본문, 발행시각, 엔티티[], 사건유형 }
```

> **왜 "검색으로 발견 → 허용분만 수집"인가?** 검색 인덱스로 URL만 찾고, robots.txt가 허용한 페이지만 실제로 가져오면 크롤링 발자국과 약관 위반 위험이 최소화됩니다. 수집한 것은 *사실*만 쓰고, 문장은 베끼지 않습니다.

> **수집 0건이면?** 그날은 건너뛰고 **비정상 종료 + 텔레그램 알림**. (조용히 빈손으로 끝내지 않음)

> **출처 설정 파일:** `config/sources.json` — 사이트별 robots/약관 검토 메모·선택자. **라이브 전에 실제 출처로 교체 필요** (현재는 플레이스홀더).

## 5. 각 단계 상세 설명

### [1] discover — 뉴스 수집
- **라이브:** Serper 뉴스 검색으로 종목별 이슈 URL 발견 → robots.txt 허용분만 Playwright로 본문 수집(불허 시 스니펫만).
- **로컬(dry-run):** `fixtureDiscover` 가 녹화된 원재료 반환 → 인터넷·키 불필요.
- 0건 → 그날 skip + 비정상 종료 + 알림.

### [2] cluster — 같은 사건 병합 (= 중복제거 B)
- **사건 지문** = `sha256(종목 | 정렬·정규화된 엔티티 | 사건유형 | 날짜버킷)`.
- 같은 지문(= 한 사건을 두 매체가 보도)을 **1개 이벤트**로 묶음 → 같은 이슈가 2개 글로 나가는 것을 원천 차단.
- 날짜버킷 24시간 → **같은 팀이라도 다른 날 경기는 병합 안 됨**(오병합 방지).

### [3] 중복제거 A — 기발행 제거
- **판단 기준(오라클) = WordPress 발행 이력(진실) + 커밋되는 원장 파일(`data/fingerprints.ndjson`).**
- 왜 둘 다? GitHub Actions 러너는 매번 새로 켜져 **로컬 저장이 사라지므로**, 레포에 커밋되는 원장으로 실행 간 기억을 유지합니다.
- 정확 지문 일치 또는 근접 텍스트 유사도 ≥ 0.85면 제거. WP 조회 실패 시 원장만으로 진행하되 **알림**.

### [4] synthesize — 다중 출처 합성
- 사용 가능 출처가 **2개 미만이면 폐기**(부적격) + 알림. **단일 출처 재작성은 절대 안 함** (저작권 핵심).
- Claude(`claude-sonnet-4-6`)가 2~3개 출처의 사실을 종합해 **우리 문체의 원본 KO 기사**를 쓰고, 각 사실을 출처에 매핑. EN은 번역.

### [5] 사실 게이트 (grounding) — 가장 중요한 안전장치
- **(1) 사실 추출:** Claude(temp 0)가 원자적 주장을 뽑음 → `[{유형, 값, 핵심여부}]`. 유형 = 스코어/날짜/선수/팀/이적료/결과/인용.
- **(2) 출처 대조(결정론적):**
  - 스코어: 출처에서 추출한 **스코어 토큰 집합에 정확히** 존재해야 함(부분문자열 매칭 금지 → `3-0`이 `13-0`에 안 걸림).
  - 이적료·날짜·숫자: 모든 숫자가 **정수 전체일치** + 단어는 **경계 일치**(→ `80 million`이 `180 million`에 안 걸림, `win`이 `winter`에 안 걸림).
  - 이름/인용: 정규화 후 출처 포함 여부.
- **(3) 판정(fail-closed):** 핵심 사실(스코어·날짜·결과·이적료·인용) 미근거가 1건이라도 있으면 → **그 이슈 차단 + 알림**. 비핵심은 제거 후 진행.

### [6] 표절 게이트
- 기사(KO+EN 본문)와 각 출처의 4-gram 유사도(Jaccard) 최대값이 `0.30` 초과면 차단.
- KO 본문만 영어 출처와 비교하면 항상 0이라 무력 → **KO·EN 본문을 함께** 비교해 EN 본문이 영어 출처와 대조되게 함.

### [7]·[8] 카드뉴스 + glyph-smoke
- VOBET의 sharp 합성 로직을 포팅. **1080×1080 정사각 카드**(텔레그램용), KO/EN 각각.
- 종목 액센트색: 축구 `#22C55E` · 야구 `#F59E0B` · 농구 `#EF4444`. 제목은 한국어 13자 / 영어 22자 단위 줄바꿈.
- **결정론적 = 항상 유효한 PNG 보장.**
- **glyph-smoke:** 제목만 투명 배경에 렌더 → 불투명 픽셀 수를 세서 **폰트 누락(두부 □□□)** 이면 전송 차단 + 알림. CI 러너(폰트 설치된 곳)에서 전송 직전 실행.

### [9] WordPress 발행
- VOBET의 검증된 발행 로직(`createWordPressPost` + Polylang 페어링 + Rank Math 메타)을 SSH/wp-cli로 호출(라이브 배선 시 포팅). KO·EN 발행 후 **번역쌍 연결**.
- **이미지 불변식(필수): 기사 1건 = 썸네일(featured) 1개 + 본문 이미지 1개.** 이미지 출처 게이트를 통과한 이미지를 WP 미디어로 업로드 → featured 지정 + 본문 상단에 1개 임베드.
  - **이미지 출처 게이트(fail-closed):** ① 화이트리스트 실제 사진(공식 소셜 임베드 / 라이선스 와이어 / Wikimedia CC, *출처표기 자동*)이 있으면 사용 → ② 없으면 **생성 이미지로 폴백.** 생성 이미지는 [7] 렌더가 **항상 유효한 PNG를 보장**하므로 불변식이 절대 깨지지 않음. (검색·다운로드·핫링크 재호스팅은 코드로 금지 — 텍스트 다중출처 합성과 같은 규율)
  - **블로그 헤더 변형:** 텔레그램 카드는 1:1(1080×1080), 블로그 featured/OG 이미지는 **16:9 변형**을 같은 `sharp` 로직으로 별도 렌더(`lib/img/cardnews.mjs` 확장). KO·EN 각각.

### [10] Telegram 배포
- `sendPhoto`(1건) / `sendMediaGroup`(다건 앨범)로 카드뉴스 전송. 경고는 `alert`(sendMessage).

### [11] 원장 기록 — 중복 영속성
- 발행+전송 성공 후 지문을 `data/fingerprints.ndjson`에 추가 → GitHub Actions가 커밋·푸시.
- **쓰기/푸시 실패 시 job 비정상 종료**(다음 실행이 옛 원장으로 중복 발행하는 것 방지).

## 5.5 차별점 파이프라인 (v0.2)

뉴스 파이프라인(위)과 **같은 fail-closed·사실게이트 규율**을 공유하되, 별도 스파인으로 동작합니다.

### ① 경기 전 인텔리전스 카드 (`pipeline/prematch.mjs`)
```
schedule(다가올 경기) → intel(폼·부상·상대전적) → "관전 포인트" 합성(LLM)
   → 사실 게이트(intel 코퍼스에 대조, fail-closed) → 프리매치 카드 렌더 → glyph-smoke
   → 텔레그램 그룹 + 개인화 푸시
```
- **카드 1장** = 매치업 + 대회/킥오프 + 양 팀 최근 폼(W/D/L 색 배지) + 핵심 부상 + "관전 포인트". (`lib/img/prematch_card.mjs`, sharp, 1080×1080)
- **합성 노트도 사실 게이트를 통과해야** 함 — 노트가 intel에 없는 부상/선수를 주장하면 **카드 차단**(뉴스 기사와 동일한 grounding 규율). dry-run에서 `fx-epl-2`가 "손흥민 결장"을 근거 없이 주장 → 차단됨을 검증.
- **데이터 출처:** 라이브는 스포츠 데이터 API(`liveIntel`/`liveSchedule`, 스텁) 또는 수집 뉴스에서 폼·부상 추출. 로컬은 fixture.
- **왜 차별적인가:** 뉴스사이트는 "벌어진 일"만 보도하고 *의미*를 패키징하지 않음. 시간 없는 독자에게 "다음 경기에서 뭘 볼지"를 한 장으로 → 고가치 트래픽 → 배너 단가↑.

### ② 개인화 팔로우 (`lib/subscriptions.mjs`, `bot/commands.mjs`)
```
사용자가 봇에 /follow Tottenham → 구독 스토어(data/subscriptions.ndjson)에 기록
   → 매일 발행/프리매치 시 매처가 이벤트 엔티티 ∩ 팔로우 계산
   → 매칭되는 구독자에게 해당 언어 카드를 개별 DM 푸시
```
- **봇 커맨드:** `/follow <팀/선수>`, `/unfollow <팀/선수>`, `/myfeed`. `processUpdate()`는 스토어 주입식이라 봇 없이 단위 테스트됨.
- **매칭:** 엔티티 정규화 후 부분문자열 매칭 — "Tottenham" 팔로우가 "Tottenham Hotspur" 이벤트에 매칭. 뉴스 카드(`core.mjs` 5i단계)와 프리매치 카드 둘 다에 적용.
- **그룹 브로드캐스트는 그대로 + 개인화는 추가 타깃 푸시.** dry-run에서 "Brazil midfielder" 팔로워가 이적 기사 DM, "Tottenham" 팔로워가 프리매치 카드 DM 받음을 검증.
- **배달 채널:** 현재는 텔레그램 DM. **후속으로 이메일 채널 추가** — 구독 회원이 관심 정보를 이메일로도 받게 함(미구현).
- **데이터 자산 (사업 핵심):** 구독·팔로우 과정에서 **이메일 + 관심사(종목/팀/선수)** 가 구독 스토어(`data/subscriptions.ndjson`, 후속으로 이메일 필드 확장)에 쌓임 → 재마케팅·배팅 배너 타깃팅의 기반.
- **왜 차별적인가:** firehose를 *내 피드*로 바꿔 "왜 계속 우리 방에 있나"의 답을 만듦 = 리텐션 + 데이터 수집.

## 13. 설계 결정 — 왜 이렇게 만들었나 (요약)

3단계 파이프라인(deep-interview → ralplan 합의 → autopilot)으로 생성됐고, 합의 단계에서 **실제 코드 대조**로 검증된 결정들:

1. **포크가 아니라 추출.** 기존 VOBET 블로그는 *키워드풀 SEO 생성기*이지 뉴스 애그리게이터가 아니다 → 검증된 하위 기능(WP·Polylang·sharp·텔레그램·cron)만 ~40-50% 추출하고, 수집·합성·중복제거·게이트 스파인은 신규 구축.
2. **다중 출처 합성이 저작권 1차 방어선** — 코드 게이트로 못 막는 위험을 구조로 막는다.
3. **사실 검증은 결정론적 출처 대조** — AI 의견투표는 환각을 통과시킨다.
4. **중복제거 영속성은 WP-진실 + 커밋 원장** — 휘발 러너에서 로컬 저장은 사라진다.
5. **모든 게이트 fail-closed** — 틀린 사실·깨진 이미지·중복을 내보내느니 건너뛴다.

> Phase-4 독립 리뷰에서 사실 게이트의 부분문자열 오탐 3건(`3-0`∈`13-0` 등)이 발견·수정되어 회귀 테스트로 고정됨. **녹색 체크는 "가짜 데이터가 우연히 피한 케이스"가 아니라 "게이트가 잡도록 설계된 케이스"를 실제로 잡는다.**
